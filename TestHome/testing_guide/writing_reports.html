Writing a Defect

Any defect created should include the following:

  • A short descriptive title
  • Information on where the bug was found/reproduced: server, build, code branch
  • An accurate description of what the bug is
  • A fully accurate and precise set of reproduction steps
  • A logserver file of the bug occurring. Even if the defect isn't showing anything in log server, or even if it is an English Language defect, you should always include a log file - it's good practice.
  • In-game screenshots/video of the bug - always in high quality PNG format and annotated appropriately

Checklist

Did you make sure your defect is not already present in the database?
Is your title accurate?
  • Does this defect have the proper prefix indicating WHERE the bug occurs?
    • Is this defect's title unique? (Meaning that no other defect could have the same name)
Did you select the project stage and build you are testing on?
  • Did you make sure you were testing the latest version?
Are the severity and priority you selected appropriate for this bug?
  • If Major/Critical, does your description explain why?
Did you select the appropriate category for this bug?
Did you check all the game locations and areas that might be affected by the bug?
  • Does your title’s prefix’s reflect the appropriate location?
Does your description follow the rules outlined in the “Defect writing” section?
  • Did you make paragraphs and are they properly spaced to ensure that the text is light and easy to read?
  • Does your description contain the information on WHERE and HOW the bug occurs?
  • Did you run your description through a spell checker?
  • Did you specify the test environment?
  • Did you include additional information about the test?
  • Have you ensured that all abbreviations used are explained in the defect?
Have you included steps for your defect?
  • Do your steps each contain a single action?
  • Are your steps numbered and in chronological order?
  • Did you filter out the steps that seem obvious or superfluous, like “Start the game”?
  • Does your last step indicate that the bug should happen at this precise moment?
Are your attachments correct?
  • Did you include all the necessary files?
Is the Fix in stage selected correctly?
Did you assign it to the proper person?
If the defect is an exploit or is describing code, did you uncheck the “Public” box?
Did you select the appropriate server (LIVE OR TEST)?

Defect Template

Title:

Outpost: Station Management button disappears

Descriptive, concise, and unique. Should contain searchable words, an appropriate prefix and should, effectively, explain the defect at a glance.

Description:

Reproducible on Singularity, EVE-2014-ARTHUR, 435082 (5/5)
Reproducible on Chaos, EVE-2014-ARTHUR, 436082 (3/3)
Non-Reproducible on Multiplicity, EVE-TRANQUILITY, 412710 (0/3) [Feature Not Present]

Should always contain server information in the above format - commas and all. Server, branch and build number are necessary, as is a reproduction rate. Repro rate should be a minimum of 3 attempts unless the defect is a text error. In the case of non-reproducible defects, you will observe that the repro rate is (0/3) indicating that you attempted to reproduce it 3 times, but the defect wasn’t seen at all. Non-repros should always follow this format.

Note the optional [ Feature Not Present ] for when servers do not have a new feature - Still be sure the repro on that server as best you can with the current system.

Character: ptw Maverick
http://87.237.38.24:50001/gm/character.py?action=Character&characterID=94604159

Always include the character name for a character that repro'd the issue and the relevant ESP page for that character on the primary test server. If, in the case, the defect is not seen on a character (e.g. on the Character Selection Screen - include the User Name and relevant ESP link.

When a player owns an outpost, they should be able to access station management. However, it was observed that if a player logs out of the server then back in, they will no longer have the "Station mgmt" button until they enter space then dock again.

Description should be clear and explain the circumstances surrounding the issue. The report should talk of “the player” and the defect should be shown as “it was observed”. Your report should not be overly verbose, but should definitively cover the defect.

[RES file paths] - If applicable
[Exception Thrown In Log Server] - If applicable (remember to remove the timestamps for clarity).

Expected Result:

The station management button should always be accessible to whoever has the outpost.

An expected result is not always needed. For instance “The game should not crash” is unnecessary. Use best judgement here and think about it from the POV of someone who may not be so EVE experienced.

Attached Files:

* Screenshot Attached
* Video Attached
* Log Sever File Attached
* DxDiag Attached

You should always list the files attached to the defect. All defects should at least contain a DxDiag and a Log Server File.

[entered by ptw]

Finally, your report should always end with this tag showing that it was entered by PTW.

Steps to reproduce:

Prerequisites:
1. Make sure you have a player who is the CEO of a corp.
2. Find a 0.0 system that has a station.

Prerequisites should cover the setup that the user needs to be in an environment to make the defect occur, but should not feature the actual steps to make the defect occur. Sometimes they are not necessary and therefore do not need to be included.

Steps to Reproduce:
1. Create a Test Territorial Claim Unit using /load me "test terr".
2. Right click on the TCU and select “Launch For Corp”.
3. Anchor the TCU then put online.
4. Using a ship with high DPS like a Phoenix with Citadel torpedo launchers, lock and fire at the station until shield, armor and structure have depleted.
5. Once the outpost has been claimed by the player corp, proceed to dock then log off the server.
6. Log back on and observe that the "Station mgmt" tab is no longer there (located on the bottom right of the station information UI).
7. Enter space then dock back again
8. Observe the station management button has reappeared.

One step to one line. If slash commands are suggested they should be added to the end of the step, not comprise the entire step. Formatting and punctuation are important! Steps should usually start with a verb and the final step should always be the defect occurring (99% of the time this will start with “Observe”). Note: If you have included prerequisites then the title “Steps to Reproduce:” should be included. If there are no pre-reqs then you can forgo including the title "Steps to repro..."

Computer:

Windows XP Professional (5.1, Build 2600)
ATI Radeon HD 5700 Series
Driver:6.14.0010.7140 (English)

This should include basic information on the machine that the defect was discovered on. OS + version, GFX card + driver is normally enough – this information can be found in your DxDiag. If, in the case of some graphics issues, you may be required to include the basic computer info of another computer on which the defect was observed. Likewise, if you are including information about a second computer – the relevant DxDiag should be included too.
